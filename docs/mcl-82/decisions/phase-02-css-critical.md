These are the seven CSS fixes committed directly to `feature/mcl-82` as defined in the implementation plan. Each item documents exactly what changed and why.

## Fix 1 — Rename `@utility bg-*` → `@utility mcl-bg-*`

**What changed**

Files:
- `src/components/manguito-theme/lib/style/mcl-theme-v4.css`
- `src/components/manguito-theme/lib/theme/static/colors.ts`
- `src/components/mcl-header/lib/mcl-header-b/NavCollapseVertical.vue`

In `mcl-theme-v4.css`:
```diff
-@utility bg-* {
+@utility mcl-bg-* {
   background-color: --alpha(--value(--color-*) / var(--mcl-bg-opacity));
 }
```

In `colors.ts`, the `bgColor` map — all 17 values updated:
```diff
  bgColor: {
-   primary: 'bg-primary',
+   primary: 'mcl-bg-primary',
-   secondary: 'bg-secondary',
+   secondary: 'mcl-bg-secondary',
    // ... same pattern for all 17 entries
  }
```

The maps `hoverBgColor`, `focusBgColor`, `beforeBgColor`, `afterBgColor`, and `bgActiveColor` are **not** changed — they generate `hover:bg-*`, `before:bg-*`, etc., which fall through to Tailwind's native utilities and are not paired with `bg-opacity-*` in practice. `@utility bg-opacity-*` is also **not** renamed — it has no Tailwind v4 collision.

In `NavCollapseVertical.vue` (line 101 of `collapseHighlightClass`):
```diff
-   'before:absolute before:w-xs before:h-full before:bg-primary before:bg-opacity-70 '
+   'before:absolute before:w-xs before:h-full before:mcl-bg-primary before:bg-opacity-70 '
```

This is the only hardcoded template location where a `before:bg-{color}` class is explicitly paired with `before:bg-opacity-*` on the same element and therefore requires the MCL opacity-aware utility.

**Why**

In Tailwind v4, `@utility bg-*` is a wildcard that matches every class beginning with `bg-`. The library defined it to provide MCL color token backgrounds with opacity support, but it silently captured and broke all of Tailwind's built-in `bg-cover`, `bg-contain`, `bg-center`, `bg-top`, `bg-gradient-to-r`, `bg-fixed`, `bg-scroll`, and other non-color `bg-*` utilities. Any consumer importing `mcl-theme-v4.css` into a Tailwind v4 project had those utilities produce empty `background-color` declarations instead of their intended effects.

Renaming to `mcl-bg-*` restricts the utility to classes that explicitly start with `mcl-bg-`, leaving Tailwind's native `bg-*` utilities untouched.

Source: audit doc `09-theme-css-audit.md` §2.1 and §P1.

**Breaking change?**

Yes. Consumers using `bg-{color}` class names generated by this library (e.g., `bg-primary`, `bg-dark-3`) must update to `mcl-bg-{color}`. Hardcoded `bg-{color}` template strings that were paired with `bg-opacity-*` also require updating.

_Changelog: The `bg-{color}` background utility classes generated by `@bobbykim/manguito-theme` have been renamed to `mcl-bg-{color}` (e.g., `mcl-bg-primary`, `mcl-bg-dark-3`). This prevents the library from overriding Tailwind v4's built-in `bg-cover`, `bg-center`, `bg-gradient-*`, and other `bg-*` utilities. Update any hardcoded `bg-{color}` class references in your templates to `mcl-bg-{color}`._

**Verification**

```bash
grep -n "@utility bg-\*" src/components/manguito-theme/lib/style/mcl-theme-v4.css
# expected: only @utility mcl-bg-* and @utility bg-opacity-*

grep -n "'bg-primary'" src/components/manguito-theme/lib/theme/static/colors.ts
# expected: no output

grep -n "before:bg-primary" src/components/mcl-header/lib/mcl-header-b/NavCollapseVertical.vue
# expected: no output
```

## Fix 2 — Rename `@utility outline-*` → `@utility mcl-outline-*`

**What changed**

Files:
- `src/components/manguito-theme/lib/style/mcl-theme-v4.css`
- `src/components/manguito-theme/lib/theme/static/colors.ts`

In `mcl-theme-v4.css`:
```diff
-@utility outline-* {
+@utility mcl-outline-* {
   outline-color: --alpha(--value(--color-*) / var(--mcl-outline-opacity));
 }
```

`@utility outline-opacity-*` is **not** renamed — it has no Tailwind v4 collision.

In `colors.ts`, the `outlineColor`, `focusOutlineColor`, and `activeOutlineColor` maps — all 17 entries each updated:
```diff
  outlineColor: {
-   primary: 'outline-primary',
+   primary: 'mcl-outline-primary',
    // ...
  }
  focusOutlineColor: {
-   primary: 'focus:outline-primary',
+   primary: 'focus:mcl-outline-primary',
    // ...
  }
  activeOutlineColor: {
-   primary: 'active:outline-primary',
+   primary: 'active:mcl-outline-primary',
    // ...
  }
```

Ring color maps (`ringColor`, `ringFocusColor`, `ringActiveColor`) are **not** changed — they use `ring-*` prefixed classes, not `outline-*`.

**Why**

Same root cause as Fix 1 but for outlines. Tailwind v4 has `outline-{width}` (e.g., `outline-2`, `outline-4`), `outline-{style}` (e.g., `outline-dashed`, `outline-dotted`), and `outline-offset-*` utilities, all of which begin with `outline-`. The library's `@utility outline-*` intercepted all of them and attempted to set `outline-color`, which produced invalid declarations for width, style, and offset classes.

Renaming to `mcl-outline-*` confines the utility to classes that explicitly start with `mcl-outline-`.

Source: audit doc `09-theme-css-audit.md` §2.2 and §P2.

**Breaking change?**

Yes. Consumers using `outline-{color}`, `focus:outline-{color}`, or `active:outline-{color}` class names generated by this library must update to `mcl-outline-{color}`, `focus:mcl-outline-{color}`, or `active:mcl-outline-{color}`.

_Changelog: The `outline-{color}` utility classes generated by `@bobbykim/manguito-theme` have been renamed to `mcl-outline-{color}` (e.g., `mcl-outline-primary`). This prevents the library from overriding Tailwind v4's built-in `outline-2`, `outline-dashed`, `outline-offset-*`, and other `outline-*` utilities. Update any hardcoded `outline-{color}` class references in your templates to `mcl-outline-{color}`._

**Verification**

```bash
grep -n "@utility outline-\*" src/components/manguito-theme/lib/style/mcl-theme-v4.css
# expected: only @utility mcl-outline-* and @utility outline-opacity-*
```

## Fix 3 — Add missing `.tooltip` qualifier to `.tooltip-bottom`

**What changed**

File: `src/components/manguito-theme/lib/style/mcl-theme-v4.css`

```diff
-.tooltip-bottom {
+.tooltip.tooltip-bottom {
   left: 50%;
   top: 110%;
   transform: translateX(-50%);
 }
```

**Why**

Every other direction variant is scoped with `.tooltip.tooltip-{direction}`:
- `.tooltip.tooltip-top { … }`
- `.tooltip.tooltip-left { … }`
- `.tooltip.tooltip-right { … }`

The bottom variant was the only one missing the `.tooltip` qualifier. Without it, any element that happened to carry the class `tooltip-bottom` — regardless of whether it was a tooltip component — received `position: absolute`, `left: 50%`, `top: 110%`, and `transform: translateX(-50%)`, causing unexpected layout breakage for unrelated elements.

Source: audit doc `09-theme-css-audit.md` §2.3 and §P3.

**Breaking change?**

No for correctly structured tooltips. The fix is a pure bugfix — it removes leaked styles from non-tooltip elements while leaving tooltip behavior identical for properly constructed tooltip elements.

**Verification**

```bash
grep -n "\.tooltip-bottom" src/components/manguito-theme/lib/style/mcl-theme-v4.css
# expected: .tooltip.tooltip-bottom only
```

## Fix 4 — Remove redundant breakpoint reset and redefinitions

**What changed**

File: `src/components/manguito-theme/lib/style/mcl-theme-v4.css`

From the `@theme {}` block, removed:
```diff
-  /* Container breakpoints */
-  --breakpoint-*: initial;
-  --breakpoint-sm: 40rem; /* 640px */
-  --breakpoint-md: 48rem; /* 768px */
-  --breakpoint-lg: 64rem; /* 1024px */
-  --breakpoint-xl: 80rem; /* 1280px */
-  --breakpoint-2xl: 96rem; /* 1536px */
```

**Why**

Tailwind v4's default breakpoints (`sm: 40rem`, `md: 48rem`, `lg: 64rem`, `xl: 80rem`, `2xl: 96rem`) are identical to the values being redefined. The redefinitions were a v3 migration shim that is no longer needed in v4.

More critically, the wildcard reset `--breakpoint-*: initial` discards any custom breakpoints a consumer might have added to their own Tailwind v4 configuration (e.g., a `xs` or `3xl` breakpoint). This was a silent consumer-hostile behaviour: importing `mcl-theme-v4.css` would wipe custom breakpoints without any error or warning.

Removing the block lets consumers' own `@theme` configurations take effect and delegates breakpoint management entirely to Tailwind v4's defaults.

Source: audit doc `09-theme-css-audit.md` §2.7.

**Breaking change?**

No. The removed values were identical to Tailwind v4's built-in defaults, so the computed breakpoints are unchanged. Consumers who were relying on the MCL library to provide these breakpoints in a non-Tailwind context should define them in their own `@theme` block.

**Verification**

```bash
grep -n "breakpoint-\*" src/components/manguito-theme/lib/style/mcl-theme-v4.css
# expected: no output
```

## Fix 5 — Replace manual `.container` with v4-native approach

**What changed**

File: `src/components/manguito-theme/lib/style/mcl-theme-v4.css`

Removed the entire `.container` block from `@layer components` (the base rule plus five `@media` breakpoint overrides):
```diff
-@layer components {
-  /* container class */
-  .container {
-    width: 100%;
-    margin-right: auto;
-    margin-left: auto;
-  }
-  @media (min-width: theme(--breakpoint-sm)) {
-    .container { max-width: theme(--breakpoint-sm); }
-  }
-  @media (min-width: theme(--breakpoint-md)) {
-    .container { max-width: theme(--breakpoint-md); }
-  }
-  @media (min-width: theme(--breakpoint-lg)) {
-    .container { max-width: theme(--breakpoint-lg); }
-  }
-  @media (min-width: theme(--breakpoint-xl)) {
-    .container { max-width: theme(--breakpoint-xl); }
-  }
-  @media (min-width: theme(--breakpoint-2xl)) {
-    .container { max-width: theme(--breakpoint-2xl); }
-  }
+@layer components {
   /* Heading size variants */
```

Added after the existing `@utility` block to preserve auto-centering behaviour:
```diff
 @utility mcl-depth-* {
   --mcl-gm-shadow: --value(integer);
 }
+
+@utility container {
+  margin-inline: auto;
+}
```

**Why**

Tailwind v4's built-in `container` utility already handles `width: 100%` and responsive `max-width` stepping from the breakpoint scale — identical behaviour to what the removed block provided. Defining `.container` in `@layer components` created a collision: the library's version competed with Tailwind's native container and could silently override consumer expectations depending on layer ordering.

The `@utility container` block re-adds only the piece Tailwind's native container does not provide by default — `margin-inline: auto` for horizontal centering — without duplicating the responsive max-width logic.

Source: audit doc `09-theme-css-audit.md` §2.6 and §3.5.

**Breaking change?**

No. The visual output is identical: `width: 100%` at all sizes with `max-width` stepping and auto-centering. Consumers who were relying on the exact `@layer components` cascade position of `.container` are unaffected because `@utility` produces equivalent specificity in v4.

**Verification**

```bash
grep -n "max-width: theme(--breakpoint" src/components/manguito-theme/lib/style/mcl-theme-v4.css
# expected: no output
```

## Fix 6 — Move tooltip caret rules to `@layer components`

**What changed**

File: `src/components/manguito-theme/lib/style/mcl-theme-v4.css`

```diff
-@layer utilities {
+@layer components {
   /* Tooltip color variants */
   /* primary */
   .tooltip.bg-primary::after { … }
   /* … 67 more rules … */
 }
```

**Why**

The 68 tooltip `::after` color rules (17 colors × 4 directions) are component-level styles — they style a named component's sub-element. Placing them in `@layer utilities` put them above `@layer components` in the Tailwind cascade (`base < components < utilities`). This gave them unintended precedence over anything a consumer placed in `@layer components`, and made the layering semantically misleading.

Moving them to `@layer components` puts them in the correct semantic position alongside the other tooltip rules defined earlier in the same layer.

Source: audit doc `09-theme-css-audit.md` §2.5 and §P5.

**Breaking change?**

No. The tooltip rendering is visually identical. Consumers who were inadvertently relying on the higher `utilities` layer precedence to have these rules win over their own `@layer components` styles may see a behaviour change, but the new behaviour is semantically correct.

**Verification**

```bash
grep -n "@layer utilities" src/components/manguito-theme/lib/style/mcl-theme-v4.css
# expected: no output
```

## Fix 7 — Fix `btn-progress-r` `transition: all`

**What changed**

File: `src/components/manguito-theme/lib/style/mcl-theme-v4.css`

In the `.btn.btn-progress.btn-progress-r::before` rule:
```diff
-.btn.btn-progress.btn-progress-r::before {
-  left: 100%;
-  transition: all 300ms linear;
-}
+.btn.btn-progress.btn-progress-r::before {
+  left: 100%;
+  transition: width 300ms linear, left 300ms linear;
+}
```

**Why**

The base `.btn.btn-progress::before` rule uses `transition: width 300ms linear` — correctly scoped to the only property being animated. The `.btn-progress-r` variant changed this to `transition: all 300ms linear`, which forces the browser to check and animate every animatable CSS property on every frame. This is a known performance anti-pattern that can cause jank and interacts poorly with other transitions applied to the same element or its children. The explicit `width 300ms linear, left 300ms linear` matches the two properties actually toggled by the `:hover`/`:focus` rules (`.btn-progress-r:hover::before` changes both `width` and `left`).

Source: audit doc `09-theme-css-audit.md` §2.10 and §P8.

**Breaking change?**

No. The animation behaviour is identical for the two properties that actually change. The only difference is that unintended cross-property transitions are eliminated.

**Verification**

```bash
grep -n "transition: all" src/components/manguito-theme/lib/style/mcl-theme-v4.css
# expected: no output
```
